## 一、緣起與目的

對話由一個具體操作問題(登出無反應)開始,逐步深入到帳號權限異常、系統根因、企業級權限架構評估,最終產出一份分階段發展規劃 + 向上級匯報報告。本紀錄完整保留整個調查歷程,包括**中途嘅誤判與糾正**,作日後查閱與決策追溯之用。

## 二、調查歷程(按提問順序)

### 1. 登出功能無反應
- **用戶提出**:點用戶設定選單撳「登出」冇反應,係咪冇登出功能?
- **AI 檢查**:`frontend/components/auth/user-menu.tsx`、`auth-provider.tsx`、`lib/auth/` 整條認證鏈
- **發現**:登出功能**存在且接線完整**;但喺**模擬登入模式**(`NEXT_PUBLIC_AUTH_MOCK=true`)下,系統會喺登出後即時自動重新登入,令登出喺畫面上似冇反應。屬模擬模式特性,非缺陷。

### 2. 註冊驗證碼點收(郵件未接)
- **用戶提出**:`/register` 要輸入驗證碼,但開發階段冇郵件功能,點收?
- **AI 檢查**:`backend/api/auth/email_provider.py`、`routes/auth.py`、`settings.py`
- **發現**:開發階段用 `ConsoleEmailProvider`(`feature_email_mock=true`,預設),**唔發真郵件**,驗證碼會**印入後端日誌 + 寫入資料庫** `users.verification_code` 欄。屬預期設計,非缺功能。

### 3. 用戶資料喺邊度查
- **用戶提出**:全部用戶資料喺邊個位置可以查看?
- **AI 檢查**:`postgres_users_store.py` schema + 直接查 `ekp-postgres` 容器
- **發現**:用戶資料喺 `ekp-postgres` → **`ekp` 資料庫** → `users` 表(同容器內 `postgres` / `langfuse` 兩個資料庫嘅 `users` 表係 Langfuse 自己嘅,**唔好撈錯**)。當時表內 0 筆,後變 1 筆(`admin@example.com`)。

### 4. 登入後係咪最大權限管理員
- **用戶提出**:登入後係咪即最大權限管理員?定有分級?
- **AI 檢查**:`schemas/rbac.py`、`models.py`、`dependency.py`、`middleware/acl.py`、`users_store.py`;並實打 `/auth/login` + `/auth/me`
- **發現**:
  - 權限**設計上分級**(管理員 / 編輯者 / 一般用戶 / 保留角色),非「一登入即最大權限」
  - 但測試帳號 `admin@example.com` **實際被解析為「只讀」(viewer)**(`role_source=default_viewer`),資料庫 `role='user'`
  - 「首位註冊用戶自動成為管理員」機制**未生效**
  - 後端跑緊**舊版程式碼**;前端「管理員」標籤**寫死**;用戶管理寫入操作**未接通**

### 5. 點解會變成噉(規劃問題定進度問題)
- **用戶提出**:係一開始冇規劃好?定規劃咗但進度未到?要由項目記錄揾根因。
- **AI 檢查**:git 版本庫考古(`schemas/rbac.py` vs `storage/rbac_storage.py` 建立時序)、W24c `progress.md`、`git show HEAD`
- **發現(關鍵糾正,見第三節)**:**唔係規劃問題,亦唔係進度問題** —— 規劃(ADR-0027)同 W24c 交付都係乾淨完整嘅;問題係**運行環境 / 檔案同步落差**(服務跑舊碼 + 本機 OneDrive 同步異常)。

### 6. 企業級夠唔夠 + 成熟系統應該點
- **用戶提出**:呢個係企業級 KM RAG,要完善 / 安全 / 精細嘅權限系統,現狀夠唔夠?成熟系統應該點?
- **AI 評估**:對標 Microsoft Purview / SharePoint、Confluence、Glean 等成熟系統九維度
- **結論**:作為 Tier 1 起步合格,作為企業級**唔夠**;**最關鍵缺口 = 檢索層文件 / 段落級存取控制**(現只到知識庫層)。AI 提出此項屬正式上線先決條件嘅觀點。

### 7. 缺口要分幾多階段開發
- **用戶提出**:SSO 可後加,其餘缺口先備齊,要分幾多 change / phase?
- **AI 產出**:P0–P6 分階段路線圖(SSO 延後),含 Tier 邊界標註。

### 8. 要唔要重做
- **用戶提出**:開始開發呢部分,係咪要重做現有嘢?
- **AI 結論**:**主要係補充擴充,唔使重做**;一次性成本 = 重建索引一次 + 小心改 AI 檢索主流程 + 授權表結構演進。

### 9. 製作計劃 + 檢查 enterprise-rbac 文件夾
- **用戶提出**:按項目規範製作完整計劃;另提到 `docs` 有 `enterprise-rbac` 文件夾,睇下係咪已有計劃。
- **AI 檢查**:git index + 工作目錄 + 規劃規範(`PROCESS.md`、模板、`ROADMAP-per-kb-tunable-config.md` 先例)
- **發現**:`enterprise-rbac` 文件夾**唔存在**,之前**冇計劃**;現有 RBAC 相關只有已完成嘅 `W24c-users-rbac/` + `ADR-0027`。

### 10. 而家係 Tier 1 定 Tier 2
- **用戶提出**:開始呢部分係咪代表 Tier 1 完成?定即將完成?(留意到 production 未上線)
- **AI 結論**:**仍係 Tier 1,未上線**(卡 Track A IT 憑證 + 真 SSO 未接);P0 屬 Tier 1 收尾,P2 安全層可能係上線先決,P3–P6 摸 Tier 2,需 ADR + Chris 拍板。

### 11. 進度百分比質疑(關鍵轉折,見第三節)
- **用戶提出**:報告嘅進度概念同我理解有差異,我認為仲係非常初期構建,你根據咩評估個百分比?
- **AI 回應**:承認百分比**量錯對象**,提出親自實測。

### 12. 實際驗證
- **用戶提出**:先做實際驗證,再改報告。
- **AI 實測**:跑測試 / 實打端點 / 查資料庫(見第四節數據)。

## 三、關鍵轉折與糾正(本次調查嘅重要特徵)

本次調查有**兩次 AI 自我糾正**,均由證據 / 用戶質疑驅動,值得紀錄:

### 糾正一:「兩套角色定義並存技術債」→ 環境同步問題
- **初判**:見到兩個檔案(`schemas/rbac.py` 三級 vs `storage/rbac_storage.py` 四級),判斷係「兩套定義並存嘅技術債」。
- **糾正依據**:`git show HEAD:backend/api/schemas/rbac.py` 顯示 HEAD 版本係**乾淨四級**(W24c F2,4 個 commit);所見嘅「三級」係**本機 / OneDrive 陳舊版本** + 後端跑舊碼,**非 HEAD 真實狀態**。
- **糾正後結論**:根因係 **deploy / restart / 同步落差**,**非設計缺陷**。

### 糾正二:「85% 完成度」→ 三層誠實量度
- **初判**:首版報告寫「後端 ~85% / 企業級深度 ~30%」。
- **用戶質疑**:認為系統仲喺非常初期,質疑百分比根據。
- **糾正依據**:承認百分比**量嘅係「程式碼存在 + 自我報告測試通過」,唔係「實際運作 / 企業級」** —— 踩中項目自身記錄嘅反模式(「偽驗收」/「擺設式」)。經親自實測後改用三層量度。
- **糾正後結論**:程式碼骨架扎實(真),但作為運作中 / 企業級系統屬早期。

## 五、過程中得出嘅結論

1. 登出 / 驗證碼「問題」均為開發模式特性,非缺陷。
2. 用戶資料在 `ekp` 資料庫 `users` 表;dev 驗證碼可由日誌 / 資料庫直接取得。
3. 權限系統設計上分三級,非「一登入即最大權限」。
4. 帳號權限異常 + 系統不穩,**根因係運行環境 / 同步落差,非規劃或設計缺陷**。
5. 程式碼骨架 + 單元測試**真實扎實**;但系統**未活、未驗證、企業核心未開始**。
6. 達到企業級**唔需要重做**,係補充擴充;但**先要基礎校正令地基活起來**。
7. 最關鍵企業缺口 = **檢索層文件級安全**,可能係正式上線先決條件。
8. 專案仍處 **Tier 1**,未上線;此次規劃橫跨 Tier 邊界,需正式決策。

## 八、最終結論

> **權限管理系統嘅程式碼地基係扎實嘅**(160 單元測試通過、13 端點、資料模型齊 —— 親自實測為證),W24c 交付係真實成果。
>
> **但系統目前未活起來** —— 實測登入失敗、角色錯誤、進階授權從未使用;而對標企業級,核心安全(檢索層文件級控制、真 SSO)仍 0%。**作為一個可運作、企業級嘅權限系統,仍處早期階段。**
>
> 根因係**運行環境 / 同步落差 + 既有債未收尾**,**非設計缺陷,唔需重做**。
>
> 前路清晰:**P0** 基礎校正令地基活起來 → **P1** 威脅模型 + 正式決策(ADR-0066)→ **P2** 檢索層文件級安全(可能係上線先決)→ P3–P6 擴充。完整路線圖已入版本庫,**待管理層審批範圍與排序。**
