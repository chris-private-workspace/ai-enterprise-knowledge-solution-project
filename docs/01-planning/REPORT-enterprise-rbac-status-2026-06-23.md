## 執行摘要

EKP 嘅權限管理系統,於 W24c 階段(2026-05-21)**搭起咗一個紮實嘅程式碼地基** —— 三級角色權限、知識庫層級授權、5 張資料表、13 個後端端點、**160 條單元測試全數通過**(本次親自實測確認,非依賴自我報告)。呢個地基係真實成果,唔係空殼。

然而,本報告必須誠實指出:**呢個地基目前仲未「活起來」**。實際運行測試顯示 —— 登入失敗、角色解析錯誤、授權資料表從未被使用過,**系統從未真正端到端走通**。而對標一個「**企業級、安全、設置精細**」嘅權限系統,核心安全能力(**檢索層文件級存取控制、真實 SSO**)仍然係 **0%**。

**核心結論**:作為一個「**可運作、已驗證、企業級**」嘅權限系統,本系統仲處於**早期階段**。但唔係從零 —— 程式碼骨架與測試係真實扎實嘅。要前進需要兩步:先做**基礎校正**(令地基活起來),再補**企業級核心安全**。完整分階段路線圖已制定並入版本庫,等待管理層審批範圍與排序。

## 一、進度與完成度(三層量度)

> **說明**:單一百分比會誤導(只反映「程式碼存在」)。下面用三把尺分層呈現,每項均有本次實測佐證。

| 量度層面 | 評估 | 實測依據 |
|---|---|---|
| **① 程式碼骨架 + 單元測試** | ✅ **扎實(已交付)** | 5 表 seed(角色 4 / 權限 92)、13 端點註冊、160 測試通過 / 0 失敗 |
| **② 端到端實際可運作** | ❌ **低(實測壞)** | 登入回 401、角色解析錯誤、授權表 0 筆紀錄 |
| **③ 企業級完整度** | ❌ **約 10–15%(早期)** | 僅有知識庫層授權骨架;檢索層文件級安全、真 SSO 等核心 = 0% |

**整體定調:作為企業級權限系統 = 早期階段。地基已搭,但系統未活、未驗證、企業核心未開始。**

## 二、實測證據(本次親自驗證,供管理層信納)

**A. 程式碼地基 — 真實扎實:**
- 資料庫 5 張 RBAC 表全部存在,種子資料已落(角色 4 個、權限矩陣 92 項)
- 後端 13 個權限相關端點全部真實註冊(經系統 API 清單確認)
- RBAC / 認證單元測試 **實跑 160 通過、8 略過、0 失敗**(88 秒)

**B. 實際運行 — 壞 / 不穩 / 未曾使用:**
- 同一管理帳號、同一密碼,數分鐘內登入由成功(200)變失敗(401)
- 登入後角色被解析為「只讀」(viewer),非預期管理員
- 帳號「已驗證」狀態在無明確操作下由否變是(系統狀態不一致)
- **群組授權 0 筆、知識庫授權 0 筆** —— 即進階授權功能**從未被實際使用過**

**C. 企業核心 — 未開始:**
- 檢索層文件 / 段落級存取控制 = 0%、真實 SSO = 0%(全程模擬)、群組成員同步 / 管理權分級 / 存取覆核 / 生命週期 = 0%、前端寫入操作(改角色 / 邀請 / 停用)= 佔位未接通

> **方法說明**:本次刻意**唔信賴開發階段嘅自我報告**(「測試通過 / 階段達標」),而係親自跑測試、實打端點、查資料庫,以免將「程式碼存在 / 階段達標」誤當「系統完成」。

## 三、系統現有內容（設計上已具備）

**權限分三級(設計):**

| 角色 | 設計上可執行範圍 |
|---|---|
| 管理員(admin) | 全部:知識庫建 / 刪 / 改 / 讀 + 管理用戶 + 派發角色 + 修改設定 + 查閱稽核 |
| 編輯者(editor) | 建立 / 編輯 / 讀取 已分配嘅知識庫 |
| 一般用戶(user) | 只讀 / 查詢 已分配嘅知識庫 |

運作機制(設計):角色記錄於用戶帳號,系統於每個受保護操作檢查角色;知識庫另有獨立授權表;認證支援三條路徑(自助註冊 / 開發模擬 / Microsoft SSO)。**注意:以上為設計能力,實際運行尚未驗證走通(見第二節 B / 第四節)。**

## 四、運行問題與根因

| 發現 | 性質 |
|---|---|
| 管理帳號實際被降為「只讀」 | 運行 / 既有債 |
| 「首位註冊用戶自動成為管理員」未生效 | 既有債,待收尾 |
| 登入行為不穩(200 → 401)、驗證狀態無聲變動 | 運行不穩 |
| 後端服務運行緊舊版程式碼(未重啟更新) | 環境同步 |
| 前端「管理員」標籤寫死;用戶管理寫入操作未接通 | 既有債,待收尾 |

> **根因說明**:經版本庫比對,**程式碼設計本身正確完整**(160 測試通過為證);問題集中喺**運行環境與檔案同步落差**(服務未更新、本機同步異常)+ **既有債未收尾**(寫入操作 / 首位管理員機制)。換言之,需要嘅係一次**基礎校正**令地基活起來,而非重做。但亦要誠實:**校正之前,系統實際上唔可用。**

## 五、與企業級嘅差距（缺口分析）

對標成熟企業權限系統(如 Microsoft Purview / SharePoint、Confluence、Glean):

| 缺口 | 重要性 | 現狀 |
|---|---|---|
| **檢索層文件 / 段落級存取控制** | 🔴 **最關鍵** | ❌ 未做(現只到知識庫層) |
| 群組成員同步 + 繼承存取 | 🟠 高 | ❌ 未做 |
| 真實 Microsoft SSO 接駁 | 🟠 高 | ❌ 仍為模擬 |
| 管理權分級(職責分立) | 🟡 中 | ❌ 現為單一管理員全權 |
| 存取覆核 / 生命週期 | 🟡 中 | ❌ 未做 |
| 屬性式權限 / 集中政策引擎 | ⚪ 規模化才需 | ❌ 未做 |

> **為何「檢索層文件級安全」最關鍵**:企業知識庫含不同敏感度文件,AI 問答若喺檢索階段撈到用戶無權查閱嘅內容,會直接摘要呈現 → 形成資料外洩。故存取控制必須喺「檢索之前 / 之中」執行。**此項對企業多用戶環境安全上線屬必備,建議視為正式上線嘅先決條件。**

## 六、發展計劃與安排（分階段路線圖）

按「依賴順序 + 風險優先 + 對系統影響」拆分。**SSO 按指示延後**。

| 階段 | 內容 | 定位 | 粗估 |
|---|---|---|---|
| **P0** | 基礎校正 + W24c 收尾(令地基活起來:修既有債、帳號權限理順、運行校正) | 基礎(可即做) | 3–5 日 |
| **P1** | 威脅模型 + 目標架構設計 + 正式決策(ADR-0066) | 設計 / 拍板點 | 5–8 日 |
| **P2** | 檢索層文件級存取控制(最關鍵安全層) | 安全核心 | 大(可分兩段) |
| **P3** | 文件 / 資料夾級細粒度授權 + 管理介面 | 擴充 | 中–大 |
| **P4** | 群組存取(成員 + 繼承) | 擴充 | 中 |
| **P5** | 管理權分級 + 存取治理(覆核 / 生命週期) | 擴充 | 中 |
| **P6**(選) | 屬性式權限 / 集中政策引擎(規模化才需) | 進階 | 大 |
| (延後) | 真實 SSO + 自動供應 + 群組真實同步 | — | 中–大 |

**已完成嘅規劃文件**(已入版本庫):整體路線圖 `ROADMAP-enterprise-rbac.md` + 首階段詳細計劃 `W88-rbac-p0-foundation/`(含交付項、驗收標準、風險、逐日安排)。

**關鍵安排原則**:影響系統結構嘅決定(P1)必須最先拍板;P2 之後任何開工,必須先有正式架構決策(ADR-0066)+ 管理層批准。

## 七、改造性質與資源考量

- **唔需要推倒重做** —— 程式碼地基扎實(160 測試為證),整體係「補充 + 往下延伸 + 加一個檢索安全層」。
- **但要先令地基活起來**(P0)—— 校正之前系統實際不可用。
- **一次性成本**:檢索索引需重建一次(資料遷移,非程式重寫)、需小心改動 AI 檢索主流程(避免影響現有問答品質)。
- **依賴**:真實 SSO 與群組真實同步,依賴 IT 部門提供環境憑證(與正式上線嘅 Track A 同一依賴)。

## 八、風險

| 風險 | 影響 | 緩解 |
|---|---|---|
| 本機檔案同步 / 開發環境不穩(本次已遇:服務跑舊碼、登入時好時壞) | 高 | P0 首件事即喺乾淨環境校正 + 驗證 |
| 真實 SSO / 上線依賴 IT 憑證(Track A) | 高 | 與 IT 部門協調時程 |
| 改動 AI 檢索主流程影響問答品質 | 中 | 逐步驗證,保留現有品質基準 |
| 範圍橫跨 Tier 邊界,需正式決策 | 中 | P1 以 ADR 拍板,避免範圍漂移 |

## 九、待決事項與建議

**需管理層 / 技術 Lead 拍板:**

1. **批准首階段(P0)開工** — 純基礎校正 + 收尾,令地基活起來。風險低,但係後續一切嘅前提。
2. **整體策略排序** — 二擇一:
   - (甲)先完成並推 Tier 1 正式上線(解 IT 憑證 + SSO),權限擴充排後;
   - (乙)將 P0 + P2(安全核心)併入「上線準備」,因企業多用戶環境冇文件級安全不宜上線。
   **建議:採(乙)思路 —— P0 即做,P2 安全層視為上線先決,經 P1 正式決策(ADR-0066)確認範圍。**
3. **資源與時程** — 確認 P1 之後嘅階段是否納入正式排期。

## 結論

> 權限管理系統嘅**程式碼地基係扎實嘅**(160 單元測試通過、13 端點、資料模型齊備 —— 本次親自實測為證),W24c 階段嘅交付係真實成果。
>
> **但必須誠實**:呢個地基目前**仲未活起來** —— 實測顯示登入失敗、角色錯誤、進階授權從未被使用,系統未曾端到端走通。而對標企業級,**核心安全能力(檢索層文件級控制、真 SSO)仍係 0%**。
>
> **故此,作為一個可運作、企業級嘅權限系統,本系統仍處於早期階段。** 前路清晰:先做基礎校正(P0)令地基活起來,再經正式決策(P1 / ADR-0066)推進企業級核心安全(P2)。完整路線圖已制定並入版本庫,**等待管理層審批範圍與排序後即可推進。**
